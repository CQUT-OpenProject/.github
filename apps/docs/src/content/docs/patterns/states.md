title: 状态与反馈
description: 加载、空状态、错误和项目成熟度表达
---

## 加载

预计耗时少于 300 ms 的操作无需展示加载动画；更长耗时的操作则应呈现与最终排版相近的骨架屏或明确的进度指示。操作按钮进入加载状态时应保持宽度不变，并临时禁用以防止重复提交。

## 空状态

空状态设计需明确区分“初次加载无内容”、“筛选过滤无结果”和“访问权限不足”等场景。页面应当提供具体原因、明确的下一步指引以及必要的帮助文档链接，避免使用大面积装饰性插图掩盖关键信息。

## 错误

错误反馈的排版结构应当包括：简明标题、具体原因、明确的下一步建议，以及在必要时附带错误代码。严禁向终端用户直接展示代码堆栈、数据库表字段或敏感的安全响应信息。

推荐：

> **课表暂时无法同步。** 知行理工服务没有响应。你可以稍后重试，已保存的本地课表不会丢失。

避免：“未知错误”“请求失败”“Error 500”。

## 项目状态

组织仓库和软件 About 页面使用以下状态词：

| 状态     | 含义                             |
| -------- | -------------------------------- |
| 实验性   | 用于验证想法，接口和数据可能变化 |
| 维护中   | 接受问题报告并计划修复           |
| 低频维护 | 只处理严重问题和必要兼容更新     |
| 已归档   | 不再维护，保留历史内容           |

项目状态标签必须附带最近确认的日期。禁止将缺少维护者的项目标注为“维护中”等绿色健康状态。

## 通知

成功提示仅用于结果确认，数秒后可自动消失；涉及错误或需要用户决策的通知则保持常驻。通知区域需合理配置 `aria-live` 属性，但应避免将频繁变化的进度更新不加过滤地播报给辅助技术。
